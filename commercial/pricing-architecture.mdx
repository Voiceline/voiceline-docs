VoiceLine pricing is built around three layers that serve different purposes. This page documents the strategic framework behind the model — why each layer is priced the way it is, how the layers interact, and the principles that govern future pricing decisions.

## The three layers

| Layer | Mechanism | What it captures |
|-------|-----------|-----------------|
| [Field Sales Fundamentals](/commercial/field-sales-fundamentals) | Per-seat | Team size — scales linearly with headcount |
| [Modules](/commercial/modules/index) + [Platform](/commercial/platform/index) | Flat monthly fee | Capability value — the inherent worth of having the functionality |
| [Credits](/commercial/credits) | Per-seat baseline + packages | Usage intensity — CRM actions automated and insights extracted |

### Layer 1: FSF — per seat

€39/user/month. Scales correctly by design: a 300-rep enterprise pays 20× more than a 15-rep SME for the same product. No changes to this mechanism.

### Layer 2: Modules and Platform — flat fees based on capability value

Module and Platform fees capture the **intrinsic value of the capability** — having pipeline visibility, coaching infrastructure, or market intelligence is worth a fixed amount regardless of team size. A Pipeline Management subscription at €400/month reflects the structural change it makes to how deals are captured and tracked, not how many reps are using it.

This means module fees should be set at the level a small company says "yes" based on the value the capability creates. The enterprise scaling is handled by Layer 3.

### Layer 3: Credits — output-based usage scaling

Credits capture **what VoiceLine produces**, not how long it takes to produce it. The right unit is the output: a CRM record created, updated, or an insight extracted.

**Why output-based, not time-based:**
- ~70% of VoiceLine interactions are memos, not assistant conversations — a minutes-based model excludes the majority of usage
- VoiceLine's product goal is to minimise rep time talking to maximise efficiency; time-based pricing creates the opposite incentive
- A 5-second voice command that creates an opportunity should cost the same as a 2-minute dictated one — the output is identical

**Why credits scale enterprise pricing correctly:**

A 15-rep SME doing moderate field work stays near the FSF baseline allowance. A 300-rep enterprise using VoiceLine intensively — visit reports, opportunity updates, case creation, analytics insights — consumes credits far beyond the baseline and purchases packages. The total bill scales with actual product usage, not an arbitrary seat-count tier.

**Additional properties of the credit model:**
- Covers all interaction modes: memo, conversation, app-triggered — input method is irrelevant
- Low adoption risk: a new module with low initial usage costs little beyond the flat fee. Revenue grows organically as rep adoption increases
- Self-adjusting revenue: as reps use modules more, credit consumption rises automatically — no renegotiation needed
- Adoption telemetry: credit consumption per output type is a direct measure of which features are generating value for each customer, enabling CS and product to act on real data

## Credit weight schedule

Credits are **weighted by output value**. A flat rate would undercharge for high-value CRM automation and overcharge for utility actions.

**Two categories of output, one shared pool:**
- **Workflow outputs** — discrete CRM records produced at the point of a rep interaction
- **Intelligence outputs** — insights extracted automatically in the background from recordings, independent of whether a workflow was completed

| Output | Credits | Rationale |
|--------|---------|-----------|
| Opportunity / lead / case created | 3 | Highest revenue impact; directly pipeline-linked |
| Visit report / interaction record created | 2 | Core FSF output; triggers downstream analytics processing |
| Account or contact record updated | 2 | CRM data quality; field-enrichment value |
| Task / reminder / email draft created | 1 | Utility; high volume but lower business impact |
| Analytics insight extracted | 1 | Background processing; scales with recording volume, independent of workflow |

A single interaction can produce multiple outputs. A memo that creates a visit report and surfaces 3 competitive signals = 2 + 3 = 5 credits. A short "create opportunity" command = 3 credits.

<Note>Credit weights and per-credit monetary rates are pending Finance validation. Published rates will appear on the [Credits](/commercial/credits) page once confirmed.</Note>

## Pricing principles

### 1. Price at realized value, not theoretical value

Starting prices must be justified by what the product demonstrably delivers today — not what the value model projects at full adoption. The theoretical ROI model is a long-term target. If we price at theoretical value before product metrics back it up, customers will downgrade or churn when the value isn't materialising.

**Implication**: Current module prices reflect conservative, early-stage value. As product metrics accumulate, they become the basis for price increase conversations — not the value model.

### 2. Modules are the growth lever

Higher ARR per customer comes from module activation — each module adds a new, self-contained revenue stream with its own value case. FSF and Platform fees are enablers, not revenue maximisers. Raising the per-seat FSF fee doesn't create new value; activating a new module does.

### 3. Prove value, then price it

The path to higher module prices is:
1. Customer activates the module
2. Product metrics show measurable outcomes (opportunities captured, insights surfaced, coaching coverage)
3. CS uses those metrics to justify the module's price in QBRs
4. When metrics are strong and consistent, a price increase is justified

Don't raise prices before this cycle completes. The module credibility test: a customer with 30 days of product data should be able to point to a measurable outcome that justifies the fee.

### 4. Procurement buys predictability

All pricing uses fixed packages — no dynamic or per-event billing. Customers need to plan annual budgets. Credits are sold as pre-purchased packages, valid for 12 months, with workspace-level pooling.

## Module value credibility

Each module has a primary measurable metric that justifies its price. These are the metrics CS teams should surface in customer conversations and QBRs.

| Module | Monthly fee | Primary credibility metric | Current strength |
|--------|------------|---------------------------|-----------------|
| Pipeline Management | €400 | Opportunities captured per rep per month that would have been missed | Strong — directly countable |
| Customer Data | €250 | Duplicates prevented + leads created from field | Strong — directly countable |
| Market Analytics Pro | €900 | Competitive signals surfaced per week across the team | Moderate — needs dashboard usage data |
| Sales Coaching Pro | €900 | Playbook coverage rate across the team | Moderate — needs coaching metric |
| Marketing Surveys | €600 | Survey completion rate vs. manual baseline | Weak — not yet live |
| Service Cases | €150 | Cases created from field vs. prior process | Weak — not yet live |

## ARR growth path

The path to higher ARR is not price increases — it's module adoption depth.

| Configuration | 30-rep team | 300-rep team |
|--------------|-------------|--------------|
| FSF only | €14,040 | €140,400 |
| + Platform Professional | €24,840 | €151,200 |
| + Market Analytics Pro | €35,640 | €162,000 |
| + Sales Coaching Pro | €46,440 | €172,800 |
| + Pipeline Management | €51,240 | €177,600 |
| + Credits (heavy usage) | ~€52,500 | ~€190,000+ |

A 300-rep enterprise on FSF + Platform + MAP + SCP + Pipeline reaches ~€177k ARR before credits. Credits add a meaningful layer at enterprise scale as usage intensity grows.
